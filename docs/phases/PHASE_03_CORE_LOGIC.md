# Phase 3: コアロジック

## 運用ルール

- ユーザーが「Phase 3 の指示書をもとに作業してください」と指示したら、この指示書を根拠に開発を開始する。
- この指示書に書かれた内容がすべて完了したら、Phase 3 は完了とし、作業を終了する。勝手に Phase 4 へ進まない。
- 次の Phase は、ユーザーが改めて実行を依頼したときに開始する。
- 判定・集計ルールはプロダクト価値の中核なので、設計書にない例外を追加しない。
- 実際のAPIキー、トークン、秘密鍵、service role keyをチャットやコミットに出さない。

## ゴール

発音判定結果をMVP仕様通りに評価・集計・出題できる純粋ロジックを実装し、ユニットテストで固定する。Phase 3 完了時点では、APIやUIから再利用できる形で、色判定、正解判定、EWMA、best attempt、復習、ストリーク、レベル、バッジ、称号、出題選定がテスト済みである。

## 前提条件

- Phase 1とPhase 2が完了している。
- DBスキーマ上のテーブル名、カラム名、音素IDが確定している。
- 参照設計書:
  - `docs/specs/02_MVP_SCOPE.md`
  - `docs/specs/04_CORE_LOGIC.md`
  - `docs/specs/05_CONTENT_SPEC.md`
  - `docs/specs/06_DATA_MODEL.md`
  - `docs/specs/10_TEST_PLAN.md`

## スコープ

含む:

- API/UIに依存しないコアロジック実装。
- 音素スコアの色判定。
- 問題正解、間違いゼロ判定。
- `phoneme_results` の最小正規化モデル定義。
- EWMA更新。
- review stageとnext review date更新。
- best attempt選定。
- 総合習熟度、ストリーク、レベル、バッジ、称号判定。
- デイリー、苦手ドリル、音素表練習の出題選定ロジック。
- ユニットテスト。

含まない:

- Azureレスポンスの実パース。
- DBへの保存処理。
- Next.js APIエンドポイント。
- Expo UI。

## 作業タスクリスト

1. ロジック配置を決める
   - APIとテストから再利用できる場所に実装する。
   - モバイルにサーバー専用処理を持ち込まない。
   - 既存のモノレポ構成に合う命名にする。

2. スコア判定を実装
   - `score >= 80` は `green`。
   - `60 <= score < 80` は `yellow`。
   - `score < 60` は `red`。
   - 境界値 `80`、`79.9`、`60`、`59.9` を必ずテストする。

3. 正解判定を実装
   - 問題の正解はターゲット音素すべてが80以上。
   - 文問題でも非ターゲット音素の低スコアは正解判定に影響させない。
   - 間違いゼロ表示は全音素が80以上。

4. EWMAと復習更新を実装
   - 初回EWMAはスコアそのまま。
   - 2回目以降は `0.3 * score + 0.7 * old`。
   - 正解時は `review_stage` を最大3まで1段進める。
   - 不正解時は `review_stage = 0`。
   - intervalはstage 0=1日、1=3日、2=7日、3=14日。
   - 複数ターゲット音素は音素単位で正解/不正解を判定する。

5. best attempt選定を実装
   - 同一問題内でターゲット音素スコア平均が最も高いattemptをbestにする。
   - 同点の場合はoverallが高いattemptを優先する。
   - さらに同点の場合は `attempt_no` が大きいattemptを優先する。

6. 集計用ロジックを実装
   - `phoneme_state` 更新対象はパック問題のターゲット音素のみ。
   - 自由入力は `phoneme_state`、snapshots、ストリーク、ヒートマップ、レベル、バッジ、称号に入れない。
   - 総合習熟度は評価済み音素のEWMA平均。未評価音素を0扱いしない。
   - ストリークは `attempts.practiced_date` のscored attemptが1件以上ある日だけを使う。

7. レベル/バッジ/称号を実装
   - レベル定義は `docs/specs/04_CORE_LOGIC.md` の表に従う。
   - バッジIDと条件は同設計書の表に従う。
   - 付与済みバッジは再付与しない。
   - 称号は優先順位の高い条件から1つだけ返す。

8. 出題選定を実装
   - デイリーは単語5、文2、合計7件。
   - 枠は苦手: 単語2+文1、新規: 単語2、復習: 単語1+文1。
   - 同一日のdaily session内では同じ `practice_item_id` を重複させない。候補不足時のみ重複を許可し、理由を返せるようにする。
   - 苦手枠、新規枠、復習枠の優先順は `docs/specs/04_CORE_LOGIC.md` に従う。
   - 苦手ドリルは苦手枠100%の変形とする。
   - 音素表練習は指定音素をターゲットに持つパック問題から選ぶ。

9. テストを作成
   - `docs/specs/10_TEST_PLAN.md` のユニットテスト項目を満たす。
   - 外部APIやDBに依存しないテストにする。

## 動作確認手順

- ユニットテストを実行し、全件成功することを確認する。
- 境界値テストが含まれていることを確認する。
- 自由入力データが集計対象外になるテストがあることを確認する。
- デイリー出題が単語5、文2、slot配分通りになることを確認する。
