# MVP Scope

## 基本方針

MVPは、iPhoneで毎日7個の発音練習を行い、音素単位の判定、短い日本語助言、進捗可視化、復習、Pro課金までを成立させる範囲に限定する。

大手アプリと競合する総合英語学習機能は入れない。会話、動画、カリキュラム、ゲーム、ランキング、シェア、シャドーイングはMVPから外す。

## MVP必須機能

### 1. デイリー練習

- 毎日7個を提示する。
- 内訳は単語5、文2。
- ホームには7個の一覧を出さない。
- スタート後、1問ずつ出題する。
- 当日初回生成時に `daily_session` として固定する。
- 7個完了で完了画面を表示し、ストリーク更新対象にする。

### 2. 出題ロジック

固定配分は次のとおり。

| 枠 | 中身 | 個数 | 選び方 |
| --- | --- | ---: | --- |
| 苦手 | 単語2 + 文1 | 3 | 習熟度EWMAが低い音素を優先。同点は日本語話者難易度が高い音素を優先。 |
| 新規 | 単語2 | 2 | 練習回数0の音素を優先。同点は日本語話者難易度が高い音素を優先。 |
| 復習 | 単語1 + 文1 | 2 | 次回復習日が到来した音素を優先。 |

新規が尽きた場合は復習に回す。全体が安定している場合は苦手枠も復習に回す。

### 3. 判定フロー

判定体験は3画面だけにする。

| 画面 | 目的 |
| --- | --- |
| 録音/判定① | 問題提示、お手本再生、録音、Azure判定、スコア、聞き比べ、2段IPA表示。 |
| 詳細② | 音素ごとの一致度、期待IPA、実測IPA、直し方への導線。 |
| 直し方③ | 混同ペアまたは音素に対応する静的な直し方、図、聞き比べ、再録音、助言評価。 |

間違いゼロ表示は、全音素が緑の場合にのみ出す。

### 4. Azure発音評価

- 判定は Azure 発音評価 API を使う。
- MVPでは prosody は使わない。
- 実測音素、期待音素、音素スコア、単語スコア、流暢さ関連レスポンスを受け取る。
- 音声は判定のため一時送信するが、サーバーには保存しない。
- Azure成功時のみ scored attempt として保存する。
- 失敗時は error log のみ保存し、練習回数、ストリーク、習熟度に入れない。

### 5. 2段IPAと色分け

- 上段に期待IPAを表示する。
- 下段に実測IPAを表示する。
- 違う音、または基準未満の音だけ色で目立たせる。
- 色のしきい値は全画面共通。

| 色 | 条件 |
| --- | --- |
| 緑 | 80以上 |
| 黄 | 60以上80未満 |
| 赤 | 60未満 |

### 6. 直し方

- MVPでは混同ペア20〜30件と汎用ページを用意する。
- 直し方は日本語で1〜2行を基本とする。
- 画像は事前生成した静的アセットを使う。
- 実行時のAI画像生成はしない。
- 頻出混同ペアはテンプレ助言を使う。
- 未知ケースのみOpenAI APIで短文整形し、結果をキャッシュする。

### 7. 進捗基礎

MVPで実装する進捗は次の3つ。

- ストリーク。
- 苦手音ヒートマップ。
- 総合習熟度の推移グラフ。

推移グラフは、生スコアの日次平均ではなく、全音素EWMAの平均である総合習熟度を使う。

### 8. レベル/バッジ/称号

MVPに含める。

- レベルは累計練習回数で決める。
- バッジは条件達成で付与する。
- 称号はユーザー状態から決める。
- 付与済みバッジは再付与しない。
- バッジ条件と称号条件は `04_CORE_LOGIC.md` で固定する。

ランキングや他ユーザー比較は含めない。

### 9. 苦手ドリル

- 練習タブに置く。
- 進捗の苦手音素から自動出題する。
- 出題配分は苦手枠100%の変形とする。
- `phoneme_state` のEWMAが低い音素を優先する。
- デイリー練習とは別セッションだが、パック問題を使う場合は `attempts` に保存し、習熟度更新対象にする。

### 10. 音素表から選ぶ練習

- 練習タブに置く。
- 音素を選ぶと、その音素をターゲットにしたパック問題を出す。
- パック問題を使うため、`attempts` に保存し、習熟度更新対象にする。

### 11. Pro限定のテキスト自由入力

- 練習タブのサブ機能として置く。
- 7日無料期間中も利用不可。
- Pro登録後のみ利用できる。
- 入力文、正規化文、IPA変換結果、OOV、判定結果は `free_attempts` に保存する。
- `phoneme_state`、復習、ヒートマップ、ストリークには一切反映しない。
- 初回利用時に保存同意を取得する。
- Proでも1日20回程度のソフトキャップを設ける。
- サーバー側レート制限も併用する。

### 12. お手本音声

- Piper をサーバーで都度生成する。
- 生成結果はキャッシュする。
- キャッシュキーはテキスト、アクセント、速度を含める。
- 音声はパックにも自由入力にも使う。
- パック音声をアプリに事前同梱しない。

### 13. 聞き比べ

- 録音/判定①と詳細②で、コーチ音声とユーザー録音を並べて再生できる。
- ユーザー録音は端末ローカルに保存できる。
- サーバーには保存しない。

### 14. 設定

MVPで設定画面に含める項目は次のとおり。

- プラン状態。
- Pro登録導線。
- 購入復元。
- 母国語: 日本語固定表示。
- 目標アクセント: US有効、UK無効表示。
- 復習リマインダーON/OFFと時刻。
- 再生速度: 標準/スロー。
- 学習データ削除。
- 学習データJSON書き出し。
- プライバシー/利用規約リンク。
- アプリバージョン。

### 15. 課金

- iOS IAP の自動更新サブスクを使う。
- RevenueCatで購読状態を管理する。
- 価格は月額580円、年額4,980円を想定する。
- 7日間はアプリ側で無料期間として管理する。
- 8日目以降は課金ウォールを表示し、Pro登録しないと練習不可にする。
- Appleのイントロオファーは使わない。
- Stripeは使わない。

### 16. 通知

- iOSローカル通知のみ。
- APNsやサーバープッシュは使わない。
- 次回復習日とユーザー設定時刻をもとに端末内で通知をスケジュールする。

### 17. データ削除と書き出し

- データ削除はMVP必須。
- JSON形式の簡易書き出しをMVP必須とする。
- 削除対象には `free_attempts`、助言評価、端末ローカル録音も含める。

## Phase 2

MVP後に検討する機能。

| 機能 | 方針 |
| --- | --- |
| 匿名ランキング | 指標は練習量とストリーク。スコア順位は使わない。 |
| 週間チャレンジ | ランキングと同じく練習量・ストリークベース。 |
| 成果シェア画像 | 端末側生成。サーバー生成しない。 |

## 将来拡張

| 機能 | 方針 |
| --- | --- |
| Android | iOS MVP後。 |
| Webお試しページ | Next.jsで将来作る候補。今回の設計対象外。 |
| 多言語UI | 世界展開時。MVPではDBに `native_language` を持つだけ。 |
| UKアクセント | データ構造だけ備え、MVPでは無効化。 |
| Stripe | 将来Web版用。iOS MVPでは使わない。 |
| prosody判定 | コスト推移を見て検討。MVPでは使わない。 |

## 明示的に作らないもの

- AI会話。
- カリキュラム。
- レッスン動画。
- 解説動画。
- 発音ゲーム。
- 単語帳。
- ネイティブそっくり訴求。
- 自信度表示。
- 効く順ランキング。
- シャドーイング。
- アプリ内での実行時AI画像生成。

## MVP受け入れ条件

MVP完了とみなすには、少なくとも次を満たす。

- 新規ユーザーが匿名認証で開始できる。
- 初回起動から7日間は、自由入力を除くMVP機能を利用できる。
- 8日目以降、Pro未登録なら練習開始時に課金ウォールが出る。
- RevenueCatでPro購読状態を取得し、購入復元できる。
- ホームからデイリー練習を開始し、1日7個を完了できる。
- 録音した音声をバックエンド経由でAzureに送り、判定結果を表示できる。
- 期待IPAと実測IPAを2段表示し、緑/黄/赤で示せる。
- 詳細②から直し方③へ遷移できる。
- attemptの全ログとbest attemptによる集計更新が保存される。
- ストリーク、ヒートマップ、推移グラフが表示される。
- レベル/バッジ/称号が条件に応じて更新される。
- 苦手ドリルが `phoneme_state` に基づき出題される。
- Proユーザーだけが自由入力を使える。
- 自由入力はデイリー集計に混入しない。
- 学習データ削除とJSON書き出しが動く。
- サーバーに音声ファイルが保存されない。

## フェーズ4セルフレビュー

- マスター設計書・既作成ドキュメントとの矛盾: Phase 2からシャドーイングを除外し、レベル/バッジ/称号をMVPに追加済み。
- Codexが実装に着手できる具体性: MVP必須、Phase 2、将来拡張、作らないもの、受け入れ条件を明確化済み。
- 用語・命名の一貫性: `daily_session`、`attempts`、`free_attempts`、`phoneme_state` を後続DB/API仕様で使う名称として固定済み。
