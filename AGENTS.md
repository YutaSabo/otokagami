# Codex作業ルール

このファイルは、リポジトリ全体でCodexを含む作業者が守るルールを定めます。

## Git運用

- `main` は、動作確認済みの正式版として扱う。
- `main` へ直接コミット・pushしない。ユーザーの明示的な許可なく `main` へマージしない。
- 作業開始前にリモートの最新状態を取得し、`main` と `origin/main` の差分を確認する。`main` を更新する必要がある場合は、作業ツリーが安全であることを確認したうえで fast-forward のみを使用する。
- 1つの依頼につき1つの作業ブランチを使用する。ブランチ名は原則として `feat/`、`fix/`、`chore/` など目的が分かる接頭辞を付ける。
- 他の依頼・作業ブランチの変更を、ユーザーの許可なく現在の作業へ混ぜない。
- 既存の未コミット変更がある場合は、それを破棄・上書きせず、由来と対象を確認してから作業する。
- `git reset --hard`、`git clean -fd`、履歴の書き換え、force pushは、ユーザーの明示的な許可なく行わない。
- ステージング前に変更一覧を確認し、安全性と依頼範囲を確認できたファイルだけを明示的に追加する。
- 中断時は、内容がまとまり、機密情報を含まない安全な変更だけをコミットして作業ブランチへpushする。未確認のファイルは残して報告する。
- 完了時は、利用可能なテスト、ビルド、静的チェックを実行し、結果を確認してからコミット・pushする。
- コード変更によって仕様、API、画面、運用手順が変わる場合は、関連するREADME、仕様書、設計書も同じ作業で更新する。

## 機密情報と生成物

- `.env`、環境別の `.env.*`、APIキー、秘密鍵、証明書、パスワード、トークン、認証情報、署名用ファイル、本番データ、ローカルDBの実データをコミットしない。
- 公開可能な環境変数テンプレートは `.env.example` を使用し、実値ではなく空値または明確なプレースホルダーだけを記載する。
- キャッシュ、依存関係、ビルド成果物、端末固有のIDE設定、OS生成ファイルはコミットしない。
- 機密情報が履歴に含まれる疑いを発見しても、勝手に履歴を書き換えない。対象ファイル、影響範囲、推奨する失効・再発行手順を値を伏せて報告する。

## 外部環境とデータベース

- ユーザーの明示的な許可なく、本番デプロイを行わない。
- ユーザーの明示的な許可なく、破壊的なデータベース操作を行わない。`supabase db reset`、本番マイグレーション、データ削除は破壊的操作として扱う。
- 外部サービスの設定変更、認証、課金が発生し得る操作は、対象と影響を事前に確認する。

## 完了報告

- 作業完了後は、変更内容、作成したブランチとコミット、pushとPull Requestの状態、テスト結果、未確認事項、ユーザーが次に行う操作を日本語で報告する。
- 削除・コミットの安全性を判断できないファイルはそのまま残し、場所と判断できない理由を報告する。
